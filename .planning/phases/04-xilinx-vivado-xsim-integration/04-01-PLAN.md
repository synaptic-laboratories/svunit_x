# Plan 04-01: Vivado xsim Certify Target and Six-Target Sign-Off

## Objective

Promote Vivado xsim from deferred future work to a real sixth certify target:

1. Wire `vivado-2025-2-1-synth-sim-full-xsim` into the flake registry using the local qualified Vivado 2025.2.1 `synth-sim-full` package.
2. Implement the `fhs` adapter in `scripts/certify.sh` so it runs a direct Vivado package smoke and then the normal SVUnit `pytest -k xsim` regression in per-fixture and compile-once modes.
3. Produce a Phase 4 sign-off manifest from a current all-six per-target run, with explicit evidence paths and jq-verifiable two-mode PASS criteria.

## Acceptance Criteria

- `nix run .#svunit-certify-vivado-2025-2-1-synth-sim-full-xsim` is exposed and passes.
- The Vivado target refuses stub inputs and requires `vivado`, `xvlog`, `xvhdl`, `xelab`, and `xsim` from the wrapper closure.
- The Vivado target writes `vivado-smoke/` evidence proving a tiny `xvlog -> xelab -> xsim` Verilog flow passed before pytest starts.
- The Vivado target writes `build-info.json` fields for `vivado_version`, `vivado_profile`, `vivado_store_path`, `vivado_qualified_root`, and `vivado_is_stub`.
- `svunit-certify-all` contains the Vivado target alongside the four Quartus targets and Verilator.
- `04-reproduce.sh` reruns all six targets into unique output directories, writes `04-sign-off-manifest.tsv`, and jq-verifies per-fixture and compile-once PASS for every row.
- `04-sign-off.md` records the two-mode six-target pass matrix, what the Vivado target covers, and what remains deferred to the future Vivado container path.

## Notes

There are two current checks, not two competing ways to invoke the target:

- The direct Vivado package smoke proves the `synth-sim-full` flake package and FHS wrappers work in the certifier closure.
- The SVUnit xsim pytest regression proves this fork's `runSVUnit -s xsim` path works with that package in both per-fixture and compile-once flows.

The future Vivado container run is a third, separate evidence layer. It should be added as a distinct target when the Xilinx flake exposes a container image.
